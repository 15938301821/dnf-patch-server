# 全局代码开发规范（NestJS + Node.js）

## 1. 适用范围与规则优先级

- 本规范适用于仓库内的 TypeScript、JavaScript、测试、脚本、配置和工程文档。
- 修改服务端代码时，必须同时遵守 `AGENTS.md` 与 `.codebuddy/rules/server.md`。
- 修改特定任务时，还需读取 `plan/jobs/<JOB-ID>/requirements.md` 和 `task.md`；任务计划不能放宽本规范或服务安全边界。
- 发生冲突时，安全不变量、经验证事实源和 `AGENTS.md` 优先；其余规则按适用范围选择更具体且更严格的要求。Server 端规范和任务计划可以收紧本规范，但不能放宽上位安全边界。
- 不得为了满足形式规则而破坏现有公开契约、数据库兼容性、安全默认值或已验证行为；发现不可兼容要求时必须停止并明确报告。

## 2. 文件规模与职责

- `.ts`、`.tsx`、`.js`、`.mjs` 单文件不得超过 500 行；该限制由 ESLint 对当前 TypeScript 和脚本代码执行。
- 文件接近 400 行时必须评估拆分，新增功能不得继续堆叠无关职责。
- 拆分应围绕领域职责、纯函数、契约、持久化或基础设施边界进行，禁止仅为规避行数限制进行机械切片。
- 不得顺带重排或格式化无关代码；优先采用满足需求的最小变更集。
- 生成目录、依赖目录、本机配置和运行产物不得作为源码维护，包括 `dist/`、`node_modules/`、`.env` 和临时数据库目录。

## 3. 中文备注与前端工程师可维护性

本仓库的主要维护者可能只熟悉前端 TypeScript。生成服务端代码时不得假设维护者熟悉
NestJS 生命周期、依赖注入、Drizzle、数据库事务、行锁、租约、outbox、对象存储或模型
凭据边界。代码和备注必须共同讲清完整业务流程，使维护者能够判断一处修改会影响哪个接口、
状态、数据表、事件和安全条件。

### 3.1 基本要求

- 新增或实质性重写的 TypeScript、JavaScript 和工程脚本必须使用中文维护备注。已有文件发生
  行为变化时，必须补齐当前变更涉及的调用链、状态变化和失败路径备注，不要求无关地重写全文件。
- 备注以“让首次接触模块的前端工程师能够安全修改”为目标，必须解释业务语义和设计原因，
  不能只翻译 TypeScript、装饰器、SQL 或 Drizzle 调用。
- 可以用前端概念帮助理解，例如把 Controller 类比为路由适配层、Service 类比为业务命令层，
  但随后必须给出准确职责，不能用类比替代事务、锁和授权语义。
- 先通过清晰命名、纵向模块、小函数、严格类型和判别联合降低理解成本，再用备注补充调用关系、
  不变量和副作用。备注不能替代混乱结构或掩盖跨层调用。
- 代码行为、参数、错误码、事务边界、锁顺序、事件顺序、鉴权方式或安全默认值变化时必须同步
  更新备注。失真、过期或与测试矛盾的备注按代码缺陷处理。
- 不使用“显然”“简单”“不用管”“永远不会失败”等措辞，也不写“完全兼容”“已经部署”
  “覆盖全部技能”或“生产可用”等超出实际证据的结论。

### 3.2 文件头元数据与维护摘要

所有新增或实质性重写的源码文件必须在 import 或可执行语句前包含以下元数据；现有文件缺失
时，不因无关的小改动批量补写：

- `@fileoverview`：文件职责、明确不负责事项，以及它在请求、后台任务或启动流程中的位置。
- `@module`：所属纵向领域模块或基础设施层。
- `@author AI生成`：固定标明 AI 参与生成。
- `@created YYYY-MM-DD`：首次创建或实质性重写日期，不得伪造历史日期。
- `@relatedPlan`：关联任务路径或任务 ID；没有独立任务时写 `N/A` 并说明直接需求。

元数据之后还必须用中文回答：

1. **调用关系**：上游由哪个 Controller、Gateway、Service、定时任务或启动入口调用，下游依赖
   哪个 Service、Repository、外部 Provider 或基础设施接口。
2. **输入输出**：主要 DTO、领域状态或数据库记录从哪里来，返回给浏览器、Worker、其他模块
   还是后台流程。
3. **副作用**：读取或写入哪些领域状态，是否开启事务、加行锁、写事件/outbox、广播、访问
   对象存储或调用模型。
4. **安全边界**：鉴权、所有权、幂等、租约、哈希、脱敏或 fail-closed 条件中哪些不能删除，
   删除后可能产生什么错误。

推荐模板：

```ts
/**
 * @fileoverview 创建 Run 的业务编排 Service；不负责 HTTP 参数提取和 Worker 本机执行。
 * @module modules/run
 * @author AI生成
 * @created YYYY-MM-DD
 * @relatedPlan N/A - 用户直接需求
 *
 * 调用关系：RunController 完成 DTO 校验后调用本 Service；本 Service 再协调 Factory、
 * Guardrail 和 RunRepository，事务提交后通知 RunGateway。
 *
 * 输入：已认证用户身份、已校验的创建 DTO 和 Idempotency-Key。
 * 输出：可返回给客户端的 Run ViewModel；不会暴露数据库行或内部凭据。
 * 副作用：在一个事务中写入 Run、Jobs、Guardrail 决策、权威事件和 outbox。
 * 安全边界：同一幂等键的不同请求必须冲突；任何 Guardrail 拒绝时不得创建 Worker Job。
 */
```

薄契约或纯类型文件也必须用一至三句说明生产方、消费方和信任边界。只做显而易见装配的
Nest Module 可以缩短正文，但必须说明它装配哪个领域能力、向外导出什么，以及不承载业务逻辑。

### 3.3 类、函数、类型与字段 JSDoc

- 所有导出的类、公共方法、函数、接口、类型和领域常量必须有中文 JSDoc。简单 DTO 可以用
  一句话说明来源和用途；有业务、安全或持久化含义的 API 必须完整说明。
- 复杂私有方法也必须有 JSDoc，特别是事务编排、状态转换、幂等、租约 fencing、哈希规范化、
  递归校验、加密、所有权检查、错误映射、重试和安全降级。
- JSDoc 按实际情况说明谁调用、何时调用、参数来源、返回语义、稳定异常、数据库读写、锁、
  广播、对象存储、模型调用和其他副作用。
- `@param` 不能只重复参数名；必须说明值来自 path/body/header/认证上下文/数据库中的哪一层、
  是否已校验、单位或范围，以及与用户、Run、Job、lease 或 attempt 的绑定关系。
- `@returns` 必须说明调用方能够依赖的业务结果，以及结果不证明什么。例如 Artifact finalized
  只证明对象长度和哈希已复核，不证明候选补丁兼容或已经部署。
- `@throws` 按稳定业务错误类别说明触发条件及 HTTP/后台状态映射；不得把数据库驱动、Node.js
  或 Provider 的所有原始错误当成公开契约。
- DTO、ViewModel、数据库行、加密持久化结构和领域输入中的非显然字段必须写属性 JSDoc。
  多个字段必须保持一致时，明确组合不变量，不能让维护者只靠相似名字猜测。

推荐模板：

```ts
/**
 * 使用当前租约完成 Worker Job，并在同一事务中聚合 Run 状态。
 *
 * 调用关系：仅由 Worker 内部 Controller 在完成 token 和 DTO 校验后调用。
 * leaseId 用于阻止旧 Worker attempt 写入已重新领取的 Job。
 *
 * @param jobId URL 中已通过 UUID schema 校验的 Job 标识。
 * @param input 当前 Worker、leaseId 和终态结果；三者必须属于同一次 claim。
 * @returns 更新后的 Job ViewModel，不包含数据库内部行或 Worker token。
 * @throws JOB_LEASE_CONFLICT 当租约所有者、编号、期限或 attempt 任一不匹配时抛出。
 */
```

### 3.4 关键业务流程的步骤备注

- 一个方法包含三个以上业务阶段，或副作用之间存在固定顺序时，必须在每个阶段前添加简短
  中文步骤备注。典型服务流程是：校验身份/所有权 -> 读取并锁定状态 -> 检查不变量 -> 写业务
  状态 -> 写权威事件/outbox -> 提交事务 -> 广播或返回。
- 步骤备注说明为什么必须按此顺序执行，以及当前步骤失败后哪些写入、广播、模型调用或 Worker
  Job 创建不得发生；不逐行解释 `if`、赋值、循环和普通 Drizzle API。
- `transaction`、`for update`、`try/catch/finally` 和提交后广播旁必须说明边界：哪些操作必须
  原子完成，锁保护哪个竞态，哪些错误会回滚，为什么广播必须等事务提交。
- 状态机分支必须说明允许的前态、目标态、拒绝非法转换的原因和重复请求行为。不能只在代码中
  堆叠状态字符串后写“更新状态”。
- 定时任务、reaper、Socket.IO 通知、模型重试和对象存储清理必须备注调度所有权、幂等前提、
  并发限制、失败后下次如何恢复，以及当前实现不支持的多实例语义。
- 安全检查旁写清防护目的，例如防跨用户访问、防旧租约写入、防幂等键重放、防任意模型外发、
  防敏感字段回显。不得只写“安全检查”或“权限验证”。

### 3.5 服务端术语首次出现时的解释

每个模块首次使用以下术语时，必须在文件头、类型 JSDoc 或相关方法备注中提供局部解释，不能
假设维护者已经读过其他模块或旧对话：

| 术语            | 备注中应说明的最小含义                                                  |
| --------------- | ----------------------------------------------------------------------- |
| Controller      | HTTP 路由适配层，只提取/校验输入并映射响应，不承载业务状态机            |
| Guard           | 请求进入 Controller 前执行的认证或访问门禁，不等于领域所有权检查        |
| DTO             | 经过运行时 schema 校验的传输结构，不等于数据库行或对外 ViewModel        |
| ViewModel       | 对调用方公开的脱敏响应结构，不直接复用持久化行                          |
| Service         | 执行业务规则和跨依赖编排的层，不直接暴露数据库细节给 Controller         |
| Repository      | 负责 Drizzle 查询、行锁和事务内持久化的模块内部边界                     |
| transaction     | 一组要么全部提交、要么全部回滚的数据库操作；提交前不能对外广播成功      |
| row lock        | 事务内阻止并发请求同时修改同一权威记录的数据库锁                        |
| idempotency     | 同一键同一请求返回同一结果，同一键不同请求必须冲突，不是忽略重复请求    |
| lease / leaseId | Worker 对 Job 的限时执行权及唯一 fencing 编号；旧编号不能完成新 attempt |
| attempt         | 同一 Job 第几次领取，Artifact、结果和终态必须绑定同一轮次               |
| outbox          | 与业务状态同事务写入、供提交后可靠通知或投递的事件记录                  |
| Guardrail       | 在创建执行任务前根据冻结策略和可验证证据作出的 fail-closed 决策         |
| Artifact        | 私有对象存储中的证据对象，数据库只保存相对引用、长度、哈希和归属元数据  |
| fail-closed     | 缺少身份、配置或证据时拒绝继续，而不是猜测、跳过校验或返回成功          |

同一文件无需重复解释同一术语，但不能只写“同上”或“见 Server 规则”。缩写首次出现时写出
中文或英文全称，例如“DTO（Data Transfer Object，数据传输对象）”。

### 3.6 Schema、配置、错误码与安全备注

- Zod schema 上方说明它保护哪个入口、由谁生产和消费、未知字段为何拒绝。非显然 refine
  必须说明阻止的错误输入、越权或资源消耗风险。
- 环境变量声明说明用途、格式、单位、范围、默认值、缺失行为和与其他变量的互斥/绑定关系；
  凭据变量必须说明不会进入响应、日志、数据库明文或测试快照。
- 稳定错误码在定义或映射处说明触发条件、HTTP 状态、调用方可采取的动作，以及是否可重试。
  不能只堆叠没有语义说明的字符串。
- 鉴权与所有权不是同一层：备注必须区分“请求带有效 token”和“当前用户拥有目标资源”。共享
  Client token、浏览器用户 token 和 Worker token 的适用入口必须写清楚。
- 加密代码说明明文何时进入/离开内存、AAD 绑定哪些身份字段、密文结构保存在哪里、轮换和
  解密失败为何 fail-closed；不得在备注示例中放入真实或看似真实的秘密。
- 模型调用说明固定角色、用户配置所有权、出站授权、超时/重试、禁用工具和 Provider 存储、
  响应校验及审计哈希，不能简写成“调用 OpenAI”。

### 3.7 测试、migration 与工程文档备注

- 测试名称使用“场景 + 预期结果”。事务、并发、鉴权、跨用户隔离、租约、幂等、凭据和
  Guardrail 回归测试必须在 Arrange 前用一至三句中文说明保护的真实风险。
- Mock、stub、fake timer、内存数据库或伪造 Provider 首次建立时说明替代哪个外部边界，以及
  本测试没有证明的真实 MySQL、对象存储、模型、Socket.IO 或 Worker 集成能力。
- 关键 `not.toHaveBeenCalled()`、回滚和零行写入断言必须能从测试名或备注看出失败后禁止发生
  的副作用。
- migration SQL 中非显然的 CHECK、限制性外键、索引和数据迁移步骤必须有 SQL 注释或配套
  migration 说明，解释保护的不变量；不得手工编辑生成文件只为增加形式注释。
- README、任务计划和 API 说明使用与代码一致的术语，明确验证范围；不得把 schema 已生成、
  Mock 通过或服务启动描述为 migration、外部模型或完整集成已验证。

## 4. 命名、格式与模块边界

- 目录使用 `kebab-case`；当前领域目录如 `artifact`、`guardrail`、`job` 保持稳定。
- TypeScript 源文件使用仓库现有的 `kebab-case.role.ts` 形式，例如 `job-reaper.service.ts`、`run-fingerprint.spec.ts`。
- 类、接口、类型和枚举使用 `PascalCase`；变量、函数和方法使用 `camelCase`；常量仅在真正不可变且具全局语义时使用 `UPPER_SNAKE_CASE`。
- 测试文件与被测文件同目录，命名为 `*.spec.ts`。
- 项目使用 ESM。TypeScript 相对导入必须沿用仓库约定的 `.js` 后缀；类型依赖使用 `import type`。
- 使用 Prettier 和 ESLint 的现有格式，不引入并行格式化规则。
- 领域模块不得导入其他模块的 repository、内部辅助函数或数据库行类型；跨模块协作只能依赖对方导出的 service、公开契约或明确的基础设施接口。
- 公共能力只能放入 `src/common/`；不得为了复用而把领域业务逻辑下沉到公共目录。

## 5. TypeScript 与运行时校验

- 禁止使用 `any`、不安全类型断言、`@ts-ignore` 或关闭严格检查来绕过类型错误。
- 外部输入从 `unknown` 开始，包括 HTTP body、query、path 参数、header、WebSocket payload、数据库 JSON、环境变量和模型响应。
- 所有外部输入必须通过 Zod 或等价的显式运行时校验后才能进入业务逻辑。
- 新增函数必须声明显式返回类型；异步函数明确返回 `Promise<T>`。
- 优先使用窄化、判别联合和纯函数；只有在运行时不变量已经被验证且无法由类型系统表达时才允许局部断言，并需说明理由。
- 可变或开放式 JSON 输入必须限制长度、深度、节点数和键范围；固定结构 DTO 则通过严格 Zod 对象限制字段与取值。不得直接信任 `Record<string, unknown>` 的内容。
- 哈希、ID、相对路径、状态和枚举值必须复用已有公共 schema，不得在各模块中形成含义不同的重复实现。

## 6. 安全、凭据与事实源

- 禁止硬编码 API Key、共享令牌、数据库密码、私人模型端点、绝对游戏路径或本机用户目录。
- 服务自身的共享令牌、数据库密码和凭据主密钥只能从进程环境或部署 Secret Manager 读取。用户 BYOK 模型密钥只允许通过受认证的专用 HTTPS 配置端点写入，明文仅可在服务进程内存中短暂存在，不得回显或进入 WebSocket、日志、错误详情、测试快照、事件、outbox 或证据正文。
- 用户 BYOK 持久化只允许保存外部 Secret Manager 引用，或由环境/KMS 主密钥保护的认证加密密文；不得保存明文、可逆编码值或数据库内自包含的解密密钥。用户凭据必须绑定稳定的持久化身份，并在读取、更新、轮换、删除及使用时校验所有权。
- 日志和错误只允许包含稳定错误码及脱敏上下文；不得输出模型原始响应、数据库驱动原始对象或完整外部载荷。
- 不得按职业名、技能名、Prompt 标题或文件名猜测 NPK、IMG、帧映射或客户端兼容性。
- `../dnf-patch` 中的根规则、职业/主题规则、manifest、Prompt、工具目录和验证报告是开发期 DNF 领域事实源；运行时只能使用经导入、审核、冻结并带来源哈希的模板和证据，不得把相邻仓库当作运行时文件依赖。
- 所有安全默认值必须采取 fail-closed：缺少证据、配置、能力注册或授权时应拒绝、阻断或降级，不得猜测后继续。
- 不得绕过 Guardrail、Worker 能力白名单、幂等约束、租约 fencing、数据库事务或限制性外键。

## 7. 变更与验证纪律

- 修改前先读取目标文件、相邻契约、调用方、测试和相关任务计划；不得仅依据文件名推断行为。
- 修复必须处理根因，并为新增分支、错误码、安全边界或回归补充对应测试。
- 数据库结构变更必须先修改 Drizzle schema，再使用 `drizzle-kit` 生成 migration；禁止手写或改写已应用的生产 migration。
- 不得修改生成产物来代替源码修复，不得把真实外部集成替换为永远成功的 mock 后宣称验证通过。
- 根据变更范围执行最小相关测试，交付前执行项目要求的门禁；失败必须继续修复或如实报告阻塞。
